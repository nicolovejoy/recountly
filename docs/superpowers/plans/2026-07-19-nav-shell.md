# Nav Shell + Library + Journal View (issue #29) — Implementation Plan

> Executed by fresh implementer + reviewer subagents per task, final branch review at
> the end. Track progress via the `- [ ]` checkboxes.

**Goal:** Replace the single scrolling page with three real routes behind a shared
bottom tab bar — `/` (Capture: recorder only), `/library` (journal cards, Unfiled,
Trash), `/search` (the existing SearchBar + results) — plus `/library/[journalId]`
(reading-order journal view) and `/library/trash` (replaces the toggle-in-
RecorderClient stopgap). URL-as-state; back button works. Design of record:
`docs/organization-and-navigation.md` (Navigation section).

**Architecture:** Same layering as everything else — SQL builders (`journal.ts`/
`entry-sql.ts`) → injectable-runner db functions (`db.ts`) → thin route handlers →
client components. Routing: a `(tabs)` route group so Capture/Library/Search share a
layout (header + tab bar + capture guard) while `/login` stays bare. Pages are thin
server components rendering client views (house pattern). Sorting is an explicit
option on the existing search/list layer (`SearchFilters.sort`), not a parallel query
path. The journal-cover task is LAST and self-contained so it can be split to a
follow-up PR if the branch gets big.

**Tech stack:** unchanged. Next.js 16 (`await params` in dynamic segments; static
segments beat dynamic — `/library/trash` wins over `/library/[journalId]`, same as
`api/entries/trash` vs `[id]`), pnpm 9 / Node 20, vitest node env.

## Global constraints

- Auth: routes copy the existing pattern verbatim — `getServerSession()` → 401
  `{ error: "Unauthorized" }`. New pages are gated by the existing `src/proxy.ts`
  matcher automatically; `isPublicPath` needs no change.
- Single user; no `user_id` anywhere.
- Suite is currently green (278 tests; run `pnpm test` for the live count) and must
  stay green after every task; new lib logic is TDD. `pnpm lint && pnpm build` green
  for every app-code task.
- Route-level integration tests (house pattern: `src/app/api/entries/trash/route.test.ts`
  — `vi.mock` auth/db, constructed `Request`s) are REQUIRED for any new/changed API
  route. NO component tests (no @testing-library in this repo).
- Schema change ONLY in Task 6 (covers): idempotent `ADD COLUMN IF NOT EXISTS` /
  `ALTER COLUMN ... DROP NOT NULL` in `db/schema.sql`.
- Reading order = `coalesce(written_at, recorded_at) ASC, recorded_at ASC`, live
  entries only. Search results stay newest-first (relevance when a query is set).
- Capture guard: while status is `connecting`/`live`/`paused`, the Library and Search
  tabs are DISABLED (rendered inert at reduced opacity) and the tab bar shows the hint
  "Recording in progress — tap Done before leaving". Chosen over a confirm dialog:
  navigation unmounts the recorder page and kills the session, so blocking is the
  simplest correct thing. Known limit (acceptable): browser back / typed URLs still
  navigate — the guard covers the tabs, which are the only in-app affordance.
- Error shape: `Response.json({ error, detail? }, { status })`.
- Journal cover photos are NOT best-effort (design doc; issue #10 is why): a failed
  cover upload fails the request loudly.

### Task 1: Data layer — journal summaries, unfiled count, sort option

**Files:**
- Modify: `src/lib/journal.ts`, `src/lib/entry-sql.ts`, `src/lib/search.ts`, `src/lib/db.ts`
- Test: `src/lib/journal.test.ts`, `src/lib/entry-sql.test.ts`, `src/lib/search.test.ts`, `src/lib/db.test.ts` (extend all)

**Interfaces — Produces:**

`src/lib/journal.ts`:

```ts
export interface JournalSummary {
  id: string;
  label: string;
  active: boolean;
  createdAt: string;
  entryCount: number;          // live entries only (deleted_at IS NULL)
  firstEntryAt: string | null; // min effective date; null when entryCount is 0
  lastEntryAt: string | null;  // max effective date
}
export function journalSummariesSql(): SqlQuery;
export function unfiledCountSql(): SqlQuery;
export function rowToJournalSummary(row: Record<string, unknown>): JournalSummary;
```

- `journalSummariesSql`: `LEFT JOIN entries e ON e.journal_id = j.id AND e.deleted_at
  IS NULL` (join-condition filter, not WHERE, so empty journals still appear with
  count 0), `count(e.id)::int AS entry_count`,
  `min/max(coalesce(e.written_at, e.recorded_at)) AS first_at/last_at`,
  `GROUP BY j.id, j.label, j.active, j.created_at`,
  `ORDER BY j.active DESC, j.created_at DESC` (same order as `listJournalsSql`).
- `unfiledCountSql`: `SELECT count(*)::int AS unfiled FROM entries WHERE journal_id
  IS NULL AND deleted_at IS NULL`.

`src/lib/entry-sql.ts`:

```ts
export type EntrySort = "newest" | "reading";
// SearchFilters gains:
//   sort?: EntrySort;  // "reading" = coalesce(written_at, recorded_at) ASC, recorded_at ASC
//   unfiled?: boolean; // true = journal_id IS NULL (mutually exclusive with journalId;
//                      // when both are set, journalId wins and unfiled is ignored)
```

- `searchEntriesSql`: when `f.sort === "reading"`, ORDER BY is
  `coalesce(written_at, recorded_at) ASC, recorded_at ASC` and overrides the rank
  expression (the journal view never sends a query; specified + tested anyway).
  Absent or `"newest"` = existing behavior, byte-for-byte (existing tests unchanged).
  When `f.unfiled` (and no `journalId`), add `journal_id IS NULL` to WHERE — a bare
  clause, no placeholder, so param numbering is untouched.

`src/lib/search.ts`:

- `parseSearchFilters` reads `?sort=` — only the literal `"reading"` is accepted
  (anything else dropped; `"newest"` is the default and never emitted); reads
  `?limit=` — integer clamped to 1..200, dropped when absent/invalid (the journal
  view asks for 200 so a full notebook isn't truncated at the default 50); reads
  `?unfiled=` — only the literal `"1"` sets `unfiled: true`, dropped when `journal`
  is also present (journalId wins).
- `buildSearchQueryString` writes `sort=reading` when `f.sort === "reading"`,
  `limit=` when set, and `unfiled=1` when `f.unfiled`.

`src/lib/db.ts`:

```ts
export async function listJournalSummaries(runner?: QueryRunner): Promise<JournalSummary[]>;
export async function countUnfiledEntries(runner?: QueryRunner): Promise<number>;
```

**Steps:**
- [x] Write failing tests: SQL text/params for both new builders (assert the
  deleted_at filter lives in the JOIN condition, not WHERE); `rowToJournalSummary`
  null-handling (0-count journal → `firstEntryAt`/`lastEntryAt` null, count coerced
  to number); `searchEntriesSql` reading order (with and without a query; with
  journalId + dates the placeholders stay sequential); parse/build round-trip for
  `sort` + `limit` (clamping, bogus values dropped); db fns via the existing
  `fakeRunner`.
- [x] Implement; `pnpm test` green.
- [x] Commit: `feat(journals): journal summaries + unfiled count + reading-order sort (#29)`

**Schema (idempotent, appended):**

```sql
-- Journal cover photos (issue #29). A photo row belongs to an entry OR a
-- journal, never both (app-enforced; no DB CHECK — a conditional constraint
-- isn't idempotently addable). entry_id drops NOT NULL for cover rows.
ALTER TABLE photos ADD COLUMN IF NOT EXISTS journal_id text REFERENCES journals(id);
ALTER TABLE photos ALTER COLUMN entry_id DROP NOT NULL;
CREATE INDEX IF NOT EXISTS photos_journal_id ON photos (journal_id);
```

**Interfaces — Produces:**

`src/lib/photo.ts`:
- `PhotoRecord.entryId: string | null`; new field `journalId: string | null`.
- `insertPhotoSql` carries 6 columns (`journal_id` appended); `rowToPhoto` maps it
  null-safe.
- `getJournalCoverSql(journalId: string): SqlQuery` — latest cover:
  `WHERE journal_id = $1 ORDER BY id DESC LIMIT 1`.

`src/lib/journal.ts`: `journalSummariesSql` gains
`(SELECT p.id FROM photos p WHERE p.journal_id = j.id ORDER BY p.id DESC LIMIT 1) AS cover_photo_id`;
`JournalSummary.coverPhotoId: string | null`; `rowToJournalSummary` maps it.

`src/lib/db.ts`: `getJournalCover(journalId: string, runner?: QueryRunner): Promise<PhotoRecord | null>`.

`POST /api/journals/[id]/cover` (multipart field `photo`): auth 401 → `await params`
→ `getJournal` 404 → validate (non-empty File, `image/*`, ≤ 8 MB — same limits as
the entries route) 400 → `uploadPhoto` (private blob; NOT best-effort — failure is
502 and no row insert) → `insertPhoto` (`entryId: null`, `journalId: id`) →
201 `{ photo: { id, url: photoProxyPath(id) } }`. Serving needs no change —
`GET /api/photo/[id]` already streams any photos row by id.

**UI:** `LibraryView` cover slot renders `<img src={`/api/photo/${coverPhotoId}`}>`
when set, placeholder otherwise. `JournalView` header gets a "Set cover" file input:
`downscalePhoto` (reuse `@/lib/image` — covers are phone photos too, same ~4.5 MB
body limit) → POST → refetch. Upload errors surface visibly (not best-effort).

**Route tests** (cover route): 401; 404 unknown journal; 400 non-image/empty; 502
when upload throws AND `insertPhoto` never called; 201 happy path inserts with
`entryId: null` + `journalId` set.
